Designing for Performance
Core Concept
  • Means improving how fast and efficiently a computer system executes programs
  • NOT just by increasing clock speed, but by improving architecture, organization and memory
Need for Performance Design
  • Cost of computers is decreasing every year
  • Demand for high performance applications increasing (image processing, speed recognition, video conferencing, simulation modeling)
  • Simply increasing clock speed is no longer enough due to power and heat limitations
How can we improve performance?
  • Increasing processor speed
  • Improve cache memory
  • Improve processor organization & architecture

Amdahl's Law
Numerical
  • Deals with the potential speed up of a program using multiple processors
  • States that the speed up due to improvement is limited by the portion of the system that cannot be improved
Speedup = 1 / [(1-f) + f/N]
fFraction of program improved
(1-f)Unaffected portion of program
NNumber of processors (factor of improvement)
✏️ Solved Example 1
// 80% of program improved by factor of 2, find speedup f = 80% = 0.8, 1-f = 0.2, N = 2 Speedup = 1 / [(1-0.8) + 0.8/2] = 1 / [0.2 + 0.4] = 1 / 0.6 = 1.66
✏️ Solved Example 2
// 40% improved by factor 2, find overall speedup f = 0.4, 1-f = 0.6, N = 2 Speedup = 1/[0.6 + 0.4/2] = 1/[0.6+0.2] = 1/0.8 = 1.25 // 80% improved by factor 4, n=4: f=0.8, N=4: 1/(0.2 + 0.8/4) = 1/(0.2+0.2) = 1/0.4 = 2.5

Chapter 3: Computer Components & Instruction Cycle

Computer Components
Definitions
  • Computer = Programmable electronic device that accepts raw data as input and processes it with a set of instructions to produce result as output
  • Input Device: allows user to enter data and instructions into computer (Keyboard, Mouse, Microphone)
  • Output Device: displays result/information through Monitor, LCD, Printer, Plotter, Speaker

  • System Unit: case that contains electronic components — sound card, video card, processor, memory

Fetch Cycle (Step by Step)
Process
  • In fetch cycle, the process fetches the next instruction from memory using the program counter
Steps
  • Step 1: PC contains address of instruction
  • Step 2: Address copied to MAR
  • Step 3: Instruction fetch from memory
  • Step 4: Instruction stored in MBR
  • Step 5: Instruction copy to IR
  • Step 6: PC increments/points to next instruction
Execution Cycle (4 types)
  • Processor-Memory (Read or write data from memory)
  • Processor-I/O Device
  • Data Processing
  • Control (change sequence of execution)

Interrupts
Important
  • Interrupt = A signal that stops the normal execution of a program so the CPU can handle an urgent work
  • Needed so CPU doesn't waste time waiting
  • Without interrupts: CPU commands I/O devices, CPU waits until device finishes → CPU time is wasted
  • With interrupts: CPU finishes instruction then does other work → CPU is not idle
Types of Interrupt
  • Program Interrupts: Caused by error in program (e.g., divide by 0, invalid instruction)
  • Timer Interrupts: Generated by internal clock, prevents one program from using CPU for long time
  • Input/Output Interrupts: Generated by I/O devices — signal sent by I/O to CPU
  • Hardware Failure Interrupts: Caused by hardware problems
Interrupt Handling Steps
  • 1. Interrupt occurs → 2. CPU finishes current instruction → 3. CPU saves current state → 4. CPU jumps to ISR → 5. ISR execute → 6. CPU restores saved state → 7. Program resumes
⚠️ ISR = Interrupt Service Routine — a special program that handles interrupts
Multiple Interrupts
  • Sequential Interrupts: Handle one after another, new interrupts wait until current ISR finishes
  • Nested Interrupts: Higher priority interrupt can interrupt a lower priority ISR. CPU switches immediately to high priority task

  • Conditional Branch: Branch depends on a condition, allows decision points
  • Arithmetic: Operations performed by the ALU
  • Address Modify: Permits address to be computed in the ALU and then inserted into instructions stored in memory

Cache Memory
Core
  • Cache is a very small, very fast memory placed between CPU and main memory
  • Reduces average memory access times as main memory is slow
Why Cache?
  • Main memory is slow
  • Cache reduces carry memory access
  • CPU is fast
  • Cache acts as buffer to reduce the speed gap
Cache Operations
  • 1. CPU sends memory address
  • 2. Cache checks if data is present
  • 3. If found → cache HIT
  • 4. If not found → cache MISS
  • 5. On miss → data fetched from main memory, stored in cache, given to CPU
Advantages
  • Reduces CPU waiting time · Improves system performance · Uses locality principle · Faster instruction execution
Limitations
  • Smaller in size · High cost · Design is very complicated

Replacement Algorithms & Write Policy
Algorithms
  • Replacement Algorithm decides which cache block to replace when cache is full
AlgorithmStrategy
LRULeast Recently Used — replaces block not used for longest time
FIFOFirst In First Out — replaces oldest block
LFULeast Frequently Used — replaces least accessed block

Write Policy
  • Decides how write operations are handled in cache memory
  • Write-Through: Data is written to both cache AND main memory simultaneously
  • Write-Back: Data is written only to cache first, and to main memory later
Principle of Locality
  • Temporal Locality: If data item is used once, it is likely to be used again soon (e.g. loop variables)
  • Spatial Locality: If memory is accessed, nearby locations are likely to be accessed (e.g. Arrays)

Levels

Cache Elements
  • Cache Addressing: Defines how CPU addresses cache memory
  • Logical Address: Cache addressed using virtual address generated by CPU. Access happens before address translation by MMU
  • Physical Address: Cache uses physical address after MMU translation
  • MMU = Memory Management Unit — hardware component that translates virtual address to physical addresses
Unified vs Split Cache
  • Unified: Uses for same instruction & data
  • Split: Separate cache for instruction and data
Cache Size Key Points
  • Larger Cache → Higher hit ratio
  • Smaller Cache → Fast access but more cache misses

Chapter 5: Internal Memory (Error Correction)

Memory types · Error correction · ECC · Hamming Code

🧠
Physical Characteristics of Memory
Types
  • Volatile Memory: Data lost when power is off. Example: RAM
  • Non-Volatile Memory: Data retains even after power off. Example: ROM
Primary vs Secondary Memory
  • Primary Memory: Very fast, data and program must be in primary memory for execution. Example: RAM, ROM
  • Secondary Memory: Computer storage device that stores data and program for a very long time
❌
Error Correction
ECC
  • Error occurs in memory because semiconductor memory is not perfect and subject to error
  • Error can happen while storing or reading data
Types of Error
  • Hard Error: Permanent physical damage in memory cells. Bit maybe stuck at 0 or 1. Causes due to manufacturing defect, environmental damage. Cannot be repaired by software
  • Soft Error: Temporary and random, alters content of one or more memory cells. Data changes without damaging memory. Causes due to power fluctuation, alpha particles, radiation. Can be detected and corrected
🛡️
Error Correcting Code (ECC) Function
Detailed
  • In ECC functions, when data is to be written into memory, a calculation function (F) is performed on data to produce a code
  • Both code and data are stored
  • An M-bit word of data + K-bit code → actual size is M+K bits
When reading:
  • Previously stored word is read out
  • Code used to detect and possibly correct error
  • New set of K-bit are generated from M-bit and compare with fetched code bits
3 Comparison Results
  • 1. No error detected → fetched data bits are sent out
  • 2. An error is detected and it is possible to correct it → data bit + error correction bit fed into corrector → produces corrected set of M-bit to be sent out
  • 3. An error is detected, it is NOT possible to correct it → condition is reported
💡 A code is characterized by the number of bit errors in a word that it can correct and detect

🫧
Bubble Sort
O(n²)Stable
  • Repeatedly compares adjacent elements and swaps if in wrong order
  • Largest element "bubbles up" to the end in each pass
  • After each pass, last sorted element is fixed
// Bubble Sort — Array: [64, 34, 25, 12, 22] for i = 0 to n-1: for j = 0 to n-i-2: if arr[j] > arr[j+1]: swap(arr[j], arr[j+1]) Pass 1: [34,25,12,22,64] ← 64 fixed Pass 2: [25,12,22,34,64] ← 34 fixed Pass 3: [12,22,25,34,64] ← sorted!
💡 Optimized: If no swap in a pass, array is sorted → O(n) best case
☝️
Selection Sort
O(n²)Unstable
  • Find minimum element from unsorted part and put at beginning
  • Divide array into sorted and unsorted parts
  • Minimum comparisons of swaps
// Array: [64, 25, 12, 22, 11] for i = 0 to n-1: minIdx = i for j = i+1 to n: if arr[j] < arr[minIdx]: minIdx = j swap(arr[i], arr[minIdx]) Pass 1: find min=11, swap pos 0 → [11,25,12,22,64] Pass 2: find min=12, swap pos 1 → [11,12,25,22,64] Result: [11,12,22,25,64]
📌
Insertion Sort
O(n) bestStable
  • Pick element and insert it into correct position in sorted part
  • Like sorting playing cards in hand
  • Efficient for small or nearly sorted arrays
// Array: [12, 11, 13, 5, 6] for i = 1 to n-1: key = arr[i] j = i - 1 while j >= 0 and arr[j] > key: arr[j+1] = arr[j]; j = j-1 arr[j+1] = key i=1: key=11, shift 12 → [11,12,13,5,6] i=2: key=13, no shift → [11,12,13,5,6] i=3: key=5, shift all → [5,11,12,13,6] i=4: key=6 → [5,6,11,12,13]
🔀
Merge Sort
O(n log n)Stable
  • Divide and Conquer algorithm
  • Divide array into two halves, sort each half, then merge
  • Always O(n log n) — best, average, worst
  • Extra space O(n) needed
// Array: [38,27,43,3,9,82,10] DIVIDE: [38,27,43,3] | [9,82,10] [38,27] | [43,3] | [9,82] | [10] [38] | [27] | [43] | [3] ... MERGE (sorted): [27,38] + [3,43] → [3,27,38,43] [9,82] + [10] → [9,10,82] [3,27,38,43] + [9,10,82] → [3,9,10,27,38,43,82]
⚡
Quick Sort
O(n log n) avgUnstable
  • Pick a pivot, partition array around pivot
  • Elements smaller than pivot go left, larger go right
  • Recursively sort left and right subarrays
  • Worst case O(n²) when already sorted and pivot is always smallest/largest
  • Best in practice for large arrays — in-place
// Array: [10,80,30,90,40,50,70], pivot=last Partition: pivot=70 [10,30,40,50] pivot=70 [80,90] ↓ Recursively sort sub-arrays Result: [10,30,40,50,70,80,90] PARTITION(arr, low, high): pivot = arr[high]; i = low - 1 for j = low to high-1: if arr[j] <= pivot: i++; swap(arr[i], arr[j]) swap(arr[i+1], arr[high]) return i+1
🏔️
Heap Sort
O(n log n)Unstable
  • Uses a Binary Heap data structure
  • Build a Max-Heap from array, then extract max repeatedly
  • Two phases: Build heap + Sort
  • Always O(n log n) for all cases
// Array: [12,11,13,5,6,7] Step 1: Build Max-Heap 13 / \ 12 7 / \ / 5 6 11 → Max-Heap Step 2: Extract max repeatedly Swap root(13) with last → [12,11,7,5,6,13] Heapify reduced heap... repeat Result: [5,6,7,11,12,13] HEAPIFY(arr, n, i): largest=i; l=2i+1; r=2i+2 if larr[largest]: largest=l if rarr[largest]: largest=r if largest!=i: swap; HEAPIFY(arr,n,largest)
🔢
Counting Sort
O(n+k)Stable
  • Non-comparison sorting — works on integers
  • Count occurrences of each element
  • k = range of input values
  • Efficient when range k is small
// Array: [1,4,1,2,7,5,2] Step 1: Count array: count[1]=2, count[2]=2, count[4]=1, count[5]=1, count[7]=1 Step 2: Cumulative count Step 3: Build output array from right to left Result: [1,1,2,2,4,5,7]
📻
Radix Sort
O(nk)Stable
  • Sort digit by digit from least significant digit (LSD) to most significant digit (MSD)
  • Uses Counting Sort as subroutine
  • Does not compare elements
// Array: [170,45,75,90,802,24,2,66] Pass 1 (units digit): [170,90,802,2,24,45,75,66] Pass 2 (tens digit): [802,2,24,45,66,170,75,90] Pass 3 (hundreds digit): [2,24,45,66,75,90,170,802]
🐚
Shell Sort
O(n log²n)Unstable
  • Generalization of Insertion Sort
  • Sort elements far apart first, gradually reduce gap (interval)
  • When gap = 1, it becomes regular insertion sort
// Array: [12,34,54,2,3], gap starts at n/2 gap = 5/2 = 2: Compare arr[0]&arr[2]: [12,34,54,2,3] → sort at gap 2 → after gap=2: [12,34,54,2,3] gap = 1 (insertion sort): Result: [2,3,12,34,54] for gap = n/2; gap > 0; gap = gap/2: for i = gap to n: key = arr[i]; j = i while j>=gap and arr[j-gap]>key: arr[j]=arr[j-gap]; j -= gap arr[j] = key

CPI (Cycles Per Instruction) — Numericals
Exam Numerical
Formula
CPI (avg) = Total Clock Cycles / Total Number of Instructions
Total Clock Cycles = Σ (Instruction Count × CPI for each type)
✏️ Solved Example
Given: Arithmetic: 400 instructions × 1 CPI = 400 cycles Load/Store: 300 instructions × 2 CPI = 600 cycles Branch: 300 instructions × 3 CPI = 900 cycles Total instructions = 1000 Total Clock Cycles = 400 + 600 + 900 = 1900 Avg CPI = 1900 / 1000 = 1.9
